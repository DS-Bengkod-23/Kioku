## 1. Pelanggaran Schema (schemas.py)

Ini adalah masalah paling serius karena Backend import langsung dari `ml.schemas`.

### 1.1 TranscriptResult — Field Hilang + Field Ekstra

**Kontrak (`ML_INTERFACE.md`):**
```python
class TranscriptResult(BaseModel):
    segments: list[TranscriptSegment]
    language: str       # required, bukan Optional
    duration: float     # WAJIB ADA
```

**Implementasi (`ml/schemas.py:12-16`):**
```python
class TranscriptResult(BaseModel):
    segments: list[TranscriptSegment]
    full_text: str        # tidak ada di kontrak
    language: Optional[str] = None  # dibuat Optional tanpa diskusi
    # duration: float   ← HILANG
```

**Masalah:**
- `duration` **tidak ada** — Backend butuh ini untuk kalkulasi "notulen sampai dalam < 50% durasi meeting" (PRD §10)
- `language` diubah dari `str` (required) menjadi `Optional[str]` tanpa persetujuan
- `full_text` adalah field tambahan yang tidak ada di kontrak (tidak fatal, tapi belum disetujui)

---

### 1.2 ActionItem — Dua Field Diganti Nama (Breaking Change)

**Kontrak (`ML_INTERFACE.md`):**
```python
class ActionItem(BaseModel):
    task: str
    assignee_name: Optional[str]   # ← nama field ini spesifik
    due_date_text: Optional[str]   # ← nama field ini spesifik
```

**Implementasi (`ml/schemas.py:18-21`):**
```python
class ActionItem(BaseModel):
    assignee: Optional[str]         # ← harusnya assignee_name
    task: str
    deadline: Optional[str] = None  # ← harusnya due_date_text
```

**Masalah:**
- `assignee_name` → `assignee`: **Breaking change** — Backend code yang sudah ditulis akan error `action_item.assignee_name` → `AttributeError`
- `due_date_text` → `deadline`: **Breaking change** — sama seperti di atas
- Rename ini juga tidak konsisten dengan `action_items.json` di golden dataset yang memakai key `"assignee"` dan `"deadline"` — artinya Azmi memakai schema-nya sendiri sebagai standar, bukan kontrak yang disepakati

---

### 1.3 SummaryResult — Seluruh Schema Salah

**Kontrak (`ML_INTERFACE.md`):**
```python
class SummaryResult(BaseModel):
    tldr: str                   # ringkasan 2-3 kalimat
    decisions: list[str]        # list keputusan yang diambil
    topics: list[str]           # topik-topik yang dibahas
```

**Implementasi (`ml/schemas.py:24-27`):**
```python
class SummaryResult(BaseModel):
    summary: str                         # ← harusnya tldr
    action_items: list[ActionItem]       # ← harusnya decisions: list[str]
    key_points: list[str]                # ← harusnya topics
```

**Masalah:**
- `tldr` → `summary`: **Breaking change**
- `decisions: list[str]` → `action_items: list[ActionItem]`: **Breaking change** — tipe data berubah total, dari list of strings menjadi list of objects. Ini mengacaukan desain karena action items harusnya di-return dari `extract_action_items()`, bukan disisipkan ke dalam `SummaryResult`
- `topics` → `key_points`: **Breaking change**
- `SummaryResult` seharusnya hanya berisi 3 field sederhana. Embedding `action_items` ke dalam summary melanggar separation of concerns yang sudah dirancang di kontrak

**Catatan:** Meskipun nama diubah, schema ini juga tidak pernah diisi dengan benar karena `extract_summary()` return `str`, bukan `SummaryResult` (lihat §2.1 di bawah).

## 2. Pelanggaran Function Signature

### 2.1 extract_summary() — Input dan Output Salah Semua

**Kontrak:**
```python
def extract_summary(transcript_text: str) -> SummaryResult:
```

**Implementasi (`ml/extract.py:17-25`):**
```python
def extract_summary(transcript: TranscriptResult) -> str:
```

**Masalah:**
- Input: seharusnya `str`, menerima `TranscriptResult` — Backend akan kirim string, langsung `TypeError`
- Output: seharusnya `SummaryResult`, mengembalikan `str` — Backend expect object dengan `.tldr`, `.decisions`, `.topics`, dapat string mentah dari LLM tanpa struktur apapun
- Ini berarti tidak ada validasi output LLM sama sekali — kalau LLM ngaco outputnya, Backend tidak tahu

---

### 2.2 extract_action_items() — Parameter Hilang, Input Salah

**Kontrak:**
```python
def extract_action_items(
    transcript_text: str,
    participant_names: list[str]
) -> list[ActionItem]:
```

**Implementasi (`ml/extract.py:28-41`):**
```python
def extract_action_items(transcript: TranscriptResult) -> list[ActionItem]:
```

**Masalah:**
- `transcript_text: str` → `transcript: TranscriptResult`: **Breaking change** — tipe input berbeda
- `participant_names: list[str]` **tidak ada** — ini bukan parameter opsional, ini adalah context yang dibutuhkan LLM untuk tahu nama-nama peserta meeting. Tanpa ini, LLM tidak bisa resolve siapa assignee-nya (lihat contoh di kontrak: `["Audi", "Helena", "Azmi"]`). Ini langsung berdampak ke akurasi action item
- Karena `participant_names` hilang, prompt di `action_items.txt` juga tidak ada mekanisme injeksi nama peserta

---

### 2.3 diarize() — Return Type Salah

**Kontrak:**
```python
def diarize(audio_path: str) -> list[TranscriptSegment]:
```

**Implementasi (`ml/diarize.py:9-22`):**
```python
def diarize(audio_path: str) -> list[dict]:
```

**Masalah:**
- Return type seharusnya `list[TranscriptSegment]` (Pydantic model), bukan `list[dict]`
- Backend yang mengikuti kontrak akan expect list of Pydantic objects, bukan raw dicts
- `merge_transcript_diarization` menerima `list[dict]` yang konsisten secara internal, tapi melanggar kontrak dengan Backend

---

### 2.4 transcribe() — Minor Issue

**Kontrak:**
```python
def transcribe(audio_path: str) -> TranscriptResult:
```

**Implementasi (`ml/transcribe.py:9`):**
```python
def transcribe(audio_path: str, model_size: str = "large-v3") -> TranscriptResult:
```

**Masalah:**
- Parameter `model_size` dengan default value tidak breaking, Backend tetap bisa panggil `transcribe(path)` tanpa masalah — ini **acceptable**
- Namun speaker awal diisi `"UNKNOWN"` (`ml/transcribe.py:17`), sementara contoh di kontrak memakai `"SPEAKER_00"`. Ini inkonsistensi minor yang tidak breaking
- Ada komentar `#coba medium dulu ya` di line 9 — ini comment development yang tidak boleh ada di production code

## 5. Prompt Engineering

### 5.1 summary.txt — Output Tidak Terstruktur

Prompt meminta "paragraf ringkas (maksimal 200 kata)" padahal `SummaryResult` (kontrak) butuh tiga field terpisah: `tldr`, `decisions`, `topics`. Tanpa structured output (JSON), Backend tidak bisa parse hasilnya ke Pydantic model.

### 5.2 Tidak Ada Injeksi participant_names

Prompt `action_items.txt` tidak menyertakan nama peserta, sehingga LLM hanya bisa menggunakan nama yang muncul di transkrip apa adanya. Kontrak sengaja mendesain `participant_names` sebagai parameter agar LLM punya daftar peserta sebagai grounding.

### 5.3 Tidak Ada Few-Shot Example

PRD §15 menyebut "few-shot" sebagai salah satu mitigasi risiko kualitas LLM lokal. Kedua prompt tidak menyertakan contoh input/output.

## 10. Catatan untuk Diskusi

Ada beberapa keputusan desain Azmi yang mungkin ada reasoning-nya tapi belum dikomunikasikan:

- **Mengapa `full_text` ditambahkan?** Kalau ini untuk kemudahan passing ke Ollama (hindari string join), itu masuk akal — tapi perlu disetujui dan ditambahkan ke kontrak
- **Mengapa `diarize()` return `list[dict]` bukan `list[TranscriptSegment]`?** Mungkin untuk menghindari circular import atau alasan performa — tapi ini tetap harus dikomunikasikan
- **Mengapa `participant_names` dihilangkan?** Mungkin belum terpikirkan cara injeksi ke prompt — perlu didiskusikan solusinya
